## Component Specs

### 1. ErrorCard

**Problem:** Every page and cockpit view implements an identical `rounded-lg border border-red-200 bg-red-50 p-4/p-6` error block inline. 12 duplicates with slight padding/wording drift.

**API design:**

```tsx
interface ErrorCardProps {
  title?: string                    // default: "Something went wrong"
  message?: string                  // free-form message; renders below title
  error?: Error | null              // extracts .message if message prop omitted
  onRetry?: () => void              // renders "Retry" button if provided
  retryLabel?: string               // default: "Retry"
  size?: 'sm' | 'md' | 'lg'        // controls padding: p-4 / p-6 / p-8
  className?: string
  // design-token placeholder: PAO Track C may rename colors to semantic tokens
  // e.g. bg-error-subtle, border-error, text-error-heading
}
```

**Variants (implicit via context, not a prop):**
- **inline** (default) ÔÇö full-width block, renders in place of content
- The `role="alert"` attribute ships on all instances for a11y

**Render contract:**
```tsx
<div role="alert" className={cn("rounded-lg border border-red-200 bg-red-50", sizeClass, className)}>
  <p className="font-semibold text-red-700">{title}</p>
  {(message ?? error?.message) && (
    <p className="mt-1 text-sm text-gray-600">{message ?? error?.message}</p>
  )}
  {onRetry && (
    <button onClick={onRetry} className="mt-3 rounded bg-blue-600 px-4 py-2 text-sm text-white hover:bg-blue-700">
      {retryLabel}
    </button>
  )}
</div>
```

**Design-token placeholder:** Once PAO Track C ships semantic color tokens (expected: `--color-error-bg`, `--color-error-border`, `--color-error-text`), update the Tailwind classes in one place in this component rather than across 12 callsites.

### 2. StatusPill

**Problem:** `LeaseStatusBadge` in `LeasesPage.tsx` is a local wrapper around shadcn `Badge`. `Badge` is also used directly in `LeaseDetailPage` and `MaintenanceWorkOrderTimeline`. The variants don't share a semantic mapping ÔÇö `success`, `outline`, `secondary` are shadcn raw variants, not domain concepts.

**Unification prerequisite:** Before promotion, unify `LeaseStatusBadge` into `StatusPill` with domain-aware variant mapping. The shadcn `Badge` can remain as the implementation substrate.

**API design:**

```tsx
type StatusPillVariant =
  | 'active'        // green ÔÇö Active leases, completed work orders
  | 'pending'       // yellow ÔÇö Pending, In Progress
  | 'expired'       // neutral/outline ÔÇö Expired leases
  | 'cancelled'     // muted red ÔÇö Cancelled
  | 'draft'         // gray ÔÇö Draft
  | 'overdue'       // red ÔÇö Overdue invoices / DaysDue > threshold

interface StatusPillProps {
  status: StatusPillVariant
  label?: string          // override display text; default: capitalize(status)
  className?: string
}
```

**Mapping to shadcn Badge variants (PAO Track C may alter):**

| StatusPill variant | shadcn Badge variant | Tailwind class override |
|---|---|---|
| active | default (mapped to green) | `bg-green-100 text-green-800 border-green-200` |
| pending | secondary (mapped to yellow) | `bg-yellow-100 text-yellow-800 border-yellow-200` |
| expired | outline | ÔÇö (default outline is fine) |
| cancelled | destructive | ÔÇö |
| draft | secondary | ÔÇö |
| overdue | destructive | ÔÇö |

**Design-token note:** PAO Track C may introduce semantic tokens for status colors (e.g., `--color-status-active`). The mapping table above becomes a token lookup at that point.

**Migration touch-list:**

| File | Change |
|---|---|
| `src/pages/LeasesPage.tsx` | Delete `LeaseStatusBadge`; use `<StatusPill status={l.status.toLowerCase() as StatusPillVariant} />` |
| `src/pages/LeaseDetailPage.tsx` | Replace raw `<Badge>` status usage with `<StatusPill>` |
| `src/components/MaintenanceWorkOrderTimeline.tsx` | Replace `<Badge>` status usage with `<StatusPill>` |

### 3. ConfirmationModal

**Problem:** `RentCollectionPage` shows an inline success panel (green bordered div) after payment recording. No shared component exists. A second instance expected in cohort-3 (JournalEntry POST confirmation per pattern-012).

**Note on name:** "Modal" is a misnomer for the current inline implementation ÔÇö it's an inline replacement panel, not a dialog overlay. Spec covers both the **inline confirmation panel** (current use case) and a future **dialog variant** (blocked on UX review). For v0.3, ship only the inline variant.

**API design:**

```tsx
interface ConfirmationPanelProps {
  title: string                       // e.g. "Payment recorded"
  children: React.ReactNode           // detail content slot
  onDismiss?: () => void              // renders dismiss/back action if provided
  dismissLabel?: string               // default: "Record another"
  actions?: React.ReactNode           // additional action slot (e.g. "View in ERPNext")
  className?: string
  // design-token placeholder: PAO Track C may introduce --color-success-bg etc.
}
```

**Render contract:**
```tsx
<div role="status" className={cn("rounded-lg border border-green-200 bg-green-50 p-6 max-w-md", className)}>
  <h2 className="text-lg font-semibold text-green-800">{title}</h2>
  <div className="mt-1 text-sm text-gray-700">{children}</div>
  {(onDismiss || actions) && (
    <div className="mt-4 flex gap-3">
      {onDismiss && (
        <button onClick={onDismiss} className="rounded bg-blue-600 px-4 py-2 text-sm text-white hover:bg-blue-700">
          {dismissLabel ?? "Record another"}
        </button>
      )}
      {actions}
    </div>
  )}
</div>
```

**Migration touch-list:**

| File | Change |
|---|---|
| `src/pages/RentCollectionPage.tsx` (confirmation panel) | Replace inline green div with `<ConfirmationPanel>` |
| `src/pages/AccountingPage.tsx` (future cohort-3) | Will use `<ConfirmationPanel>` for JournalEntry confirmation |

**Future v0.4 dialog variant:** A `ConfirmationDialog` wrapping shadcn `Dialog` for use cases where the confirmation should overlay rather than replace content. Blocked on: (a) UX decision on which flows warrant dialog vs inline; (b) PAO Track C design tokens.

### 4. LoadingSkeleton

**Problem:** Loading states use text-only divs (`"Loading leasesÔÇª"`, `"LoadingÔÇª"`) ÔÇö no visual skeleton animation. Shadcn ships `Skeleton` (`animate-pulse rounded-md bg-muted`). Standardize on shadcn Skeleton throughout.

**Not a new component** ÔÇö this is a usage convention + shadcn import, not a new `@sunfish/ui-react` export. `Skeleton` already ships in shadcn's component registry; just ensure it's imported from the correct path and used consistently.

**Convention:**

```tsx
import { Skeleton } from '@/components/ui/skeleton'

// List loading state
<div className="space-y-3">
  {Array.from({ length: 5 }).map((_, i) => (
    <Skeleton key={i} className="h-12 w-full rounded-lg" />
  ))}
</div>

// Form/detail loading state
<div className="space-y-4">
  <Skeleton className="h-6 w-48" />   // heading
  <Skeleton className="h-4 w-full" /> // body line
  <Skeleton className="h-4 w-3/4" />  // body line shorter
</div>
```

**Migration touch-list (replace all text-only loading divs):**
