## §4 — Devil's Advocate

**Stance:** Was extending the existing enum the right architectural call? Should this have been a new sibling enum?

### DA-1 — Additive extension vs new sibling enum (`FeatureSequestrationFlagKind`)

**The contention:** instead of adding `FeatureGateOff` as the 6th value of `SequestrationFlagKind`, the author could have introduced a new sibling enum `FeatureSequestrationFlagKind` (with one value: `FeatureGateOff`) and a discriminator field on `SequesteredRecord` (e.g., `Cause: SequestrationCause` where `SequestrationCause` is a tagged union of `FormFactor(SequestrationFlagKind)` + `Feature(FeatureSequestrationFlagKind)` + future causes).

**Council analysis:**

- **Sibling-enum is over-engineered for the current problem.** The existing `SequestrationFlagKind` is already a flat enum that mixes 4 form-factor-capability cases with 1 quorum-eligibility case (`FormFactorQuorumIneligible`) — i.e., the enum is **already semantically broader than its name implies**. Adding `FeatureGateOff` is consistent with the existing precedent of "this enum names every reason a record might be in the sequestration partition, including ones that aren't strictly form-factor-capability-driven."
- **Wire-form simplicity wins.** `SequesteredRecord.Flag` is a single nullable enum on the wire; introducing a discriminator + sibling-enum would force a schema migration on existing persisted rows AND ripple through `MigrationAuditPayloads.Sequestered` + `ISequestrationStore.SequesterAsync` + `InvariantDlfTests` per-value pins. The author's choice avoids all that ripple.
- **Future evolution path is preserved.** If/when the cohort grows past 7-8 values and the enum starts feeling overloaded, a future amendment can introduce the sibling-enum pattern with a discriminator AND keep `SequestrationFlagKind` as the legacy single-enum surface for back-compat (the W#35 substrate handles this gracefully because the wire form is literal-name, not ordinal).
- **Audit-by-construction is satisfied either way.** Both designs answer "why is this record sequestered?" from the audit substrate alone; the additive-extension is just the simpler-of-two satisfying-designs.

**Devil's-Advocate verdict on DA-1:** the author's choice is correct. Additive extension is right-sized for the current cohort.

### DA-2 — Cross-package coupling (ADR 0075 → W#35 enum)

**The contention:** A11 lands the substrate enum value, but ADR 0075 (W#44) is the only consumer that produces `FeatureGateOff`. Is the cross-package coupling correctly handled, or does A11 leave the substrate with a "dangling" enum value that no one writes?

**Council analysis:**

- **Pre-A11, the enum has zero unused values** — every value is produced by `InMemoryFormFactorMigrationService.ClassifySequestrationFlag` (verified at lines 288-311). A11 will introduce the first unused-by-the-substrate-itself value.
- **This is intentional and correct.** The W#35 substrate is **the contract surface for sequestration**, not the only producer. ADR 0075 is a consumer that reuses the substrate's contract for a different cause-class; that's the substrate's job. The substrate doesn't need to know who produces `FeatureGateOff`; it just needs to expose the right contract value so consumers can write it.
- **Documentation discipline is maintained.** The XML doc on `FeatureGateOff` explicitly says "Per ADR 0028-A11 + ADR 0075" — a future maintainer reading the enum sees the cross-reference and can find the consumer.
- **Sequencing is correct.** A11 lands first; ADR 0075 then references the now-existing enum value in its §Decision; the W#44 implementation in `Sunfish.Foundation.ExtensionFields` (or wherever the consumer lives) writes `SequestrationFlagKind.FeatureGateOff` into `SequesteredRecord.Flag` via `ISequestrationStore.SequesterAsync`.

**Devil's-Advocate verdict on DA-2:** cross-package coupling is correctly handled. A11 ships the substrate change; ADR 0075's separate amendment ships the consumer wiring; the two are coordinated via the council recommendation chain.

### DA-3 — Should A11 also add an `AuditEventType.FeatureGateOff` value?

**The contention:** the amendment introduces `SequestrationFlagKind.FeatureGateOff` but explicitly does NOT add a corresponding `AuditEventType.FeatureGateOff`. §A11.4 says:

> A new audit-event-type for `FeatureGateOff` is NOT added in A11. The W#44 consumer (ADR 0075) will register its own audit-event-type; the existing `AuditEventType.PlaintextSequestered` / `AuditEventType.CiphertextSequestered` event-type taxonomy is form-factor-driven and stays form-factor-driven. This is intentional separation: the **flag** carries the cause; the **audit-event-type** reflects the substrate that emitted the audit row.

**Council analysis:**

- **The flag-vs-event-type separation is principled.** `SequestrationFlagKind` is the cause-classifier; `AuditEventType` is the substrate-emitter-classifier. The existing `AuditEventType` enum in W#35 is W#35's own substrate's emitter taxonomy; adding `FeatureGateOff` to it would conflate W#44's emitter (the ExtensionFields feature-evaluation hook) with W#35's emitter (the form-factor migration service).
- **The right place to add `AuditEventType.FeatureGateOff` is in W#44**, not W#35. If the W#44 ADR 0075 amendment chooses to add an event-type, it should live in the consumer's emitter taxonomy, not in the substrate's. (Alternatively, W#44 may choose to use the existing `AuditEventType.PlaintextSequestered` for the audit row's event-type while still writing `SequestrationFlagKind.FeatureGateOff` for the flag — i.e., "the substrate emitted a sequestration audit row; the cause was feature-gate-off." This is a W#44 decision.)
- A11 is correct to leave this open and route the decision to the W#44 amendment council.

**Devil's-Advocate verdict on DA-3:** the flag-vs-event-type separation is the right boundary. No A11 change needed.

### DA-4 — Is the §A0 self-audit "complete"? Could anything still slip through pre-merge?

**The contention:** §A0 self-audits have caught only ~65% of structural-citation failures across the cohort. Could A11 still have an undetected drift?

**Council analysis:** I executed the full 3-direction sweep (§3 above) and found 0 drift. The full surface of A11's citations is small enough (8 cited symbols + 3 cited claims + 7 cited paths) that complete coverage is achievable, and the author achieved it. **The §A0.5 / §A11.5 self-audit caught everything; the council's verification confirms nothing slipped through.**

**Devil's-Advocate verdict on DA-4:** A11's §A0 caught everything. This is the cohort's first counter-example to the "A0 catches ~65%" pattern — though the sample size of 1 doesn't disprove the cohort metric for larger amendments.
