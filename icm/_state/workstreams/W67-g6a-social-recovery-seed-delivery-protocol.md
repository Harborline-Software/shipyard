## Notes

**Context.** The G7 conformance baseline scan (2026-05-16) confirmed G6 is PARTIAL — the social recovery state machine, hosting infrastructure, and 5 Razor pages are wired, but `AnchorRecoveryCompletionHandler.HandleAsync` stubs the SQLCipher rekey:

```csharp
// TODO (post-W#63): IEncryptedStore.RotateKeyAsync(...)
//                   IAuditTrail.AppendAsync(new AuditRecord(...))
//                   ISyncDaemon.AnnounceIdentityRotation(...)
```

**What the G7 report got wrong.** The filed report (committed at `6c262a69`) incorrectly attributes the stub to `IEncryptedStore.RotateKeyAsync` not existing. It does exist:
- `packages/foundation-localfirst/Encryption/IEncryptedStore.cs` line 73: `RotateKeyAsync(ReadOnlyMemory<byte> newKey, ct)` — virtual, throws NotSupportedException by default
- `packages/foundation-localfirst/Encryption/SqlCipherEncryptedStore.cs` line 209: overrides via `PRAGMA rekey`

**True root cause — protocol design gap.** `AnchorRecoveryCompletionHandler` cannot call `RotateKeyAsync` because it has no `newKey` bytes. The `RecoveryCompleted` event only carries:
- `grace.startedAt` (ISO-8601 timestamp)
- `grace.elapsedAt` (grace window end timestamp)
- `attestations.count` (integer)

No seed material, no key bytes, no encrypted payload. The Phase 1 social recovery protocol (#48a) proves that a quorum of trustees RECOGNIZES the new device's identity — it does not DELIVER key material to the new device.

**Why `RecoveryRequest.EphemeralPublicKey` exists.** The `RecoveryRequest` record contains an `EphemeralPublicKey` (32 bytes, Ed25519) generated by the recovering device. This is the design hook for key delivery: trustees SHOULD encrypt seed material using this key so the recovering device can decrypt it with the corresponding private key. But:
- `TrusteeAttestation` has no `EncryptedSeedShare` field (or equivalent)
- The `TrusteeSetupPage.razor` has no flow to distribute seed shares to trustees

**What Phase 2 needs to define** (XO research scope):

1. **Trustee setup flow.** When the owner designates trustees, the owner's root seed (or Shamir shares of it) must be distributed to trustees in a way trustees can later re-encrypt for a recovering device. Options:
   - Shamir (k-of-n threshold): owner splits 32-byte root seed into 5 shares; each trustee receives 1 encrypted share; reconstruction requires ≥3 shares — matches the 3-of-5 quorum model exactly.
   - Full-copy (simpler): each trustee receives the full root seed encrypted under their identity key; any attesting trustee can provide the seed. Weaker (any single trustee has full access), but simpler.

2. **Attestation payload.** `TrusteeAttestation` needs an `EncryptedSeedShare: byte[]` field:
   - Trustee decrypts their stored seed share using their identity key
   - Re-encrypts the share using ECIES / X25519 DH with the recovering device's `EphemeralPublicKey` from `RecoveryRequest`
   - Includes the ciphertext in the attestation

3. **Recovery coordinator changes.** After grace expires, `EvaluateGracePeriodAsync` must provide the collected attestation payloads to the completion handler — currently the coordinator discards them after quorum counting.

4. **Completion handler.** After collecting ≥3 encrypted shares, `AnchorRecoveryCompletionHandler`:
   - Decrypts shares using the ephemeral private key (held in memory on the recovering device's session)
   - Reconstructs the root seed via Shamir (or decrypts directly if full-copy design)
   - Calls `IEncryptedStore.RotateKeyAsync(newSqlCipherKey)` where `newSqlCipherKey` is derived from the reconstructed root seed via the standard `SqlCipherKeyDerivation` path
   - Emits the `RecoveryRekey` audit record

5. **ADR amendment.** ADR 0046 Phase 1 is silent on the seed delivery protocol. An amendment (A6 or new ADR) must specify:
   - Shamir vs. full-copy choice with threat model
   - Wire format for trustee-stored encrypted seed (trustee setup payload)
   - Wire format for `TrusteeAttestation.EncryptedSeedShare`
   - X25519 key agreement + HKDF-SHA256 + AES-GCM for the ECIES envelope
   - Ephemeral private key lifetime on the recovering device (session-only; never persisted)

**Scope.** XO research session: ~3-4h. Outputs: ADR 0046 amendment (A6) specifying the protocol; updated `TrusteeAttestation` and `RecoveryRequest` context; W#67 → `ready-to-build` once council-reviewed.

**Unblocks.** Full G6 closure — `AnchorRecoveryCompletionHandler.HandleAsync` wiring the real rekey path.
